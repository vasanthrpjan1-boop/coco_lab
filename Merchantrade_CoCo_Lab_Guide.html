The Business Challenge

Merchantrade spends around RM100 to acquire every telco customer. With competitors driving aggressive pricing into the market, the payback period keeps stretching. The only sustainable way to win is to maximise the lifetime value (LTV) of customers Merchantrade already has — by cross-selling ecosystem products: insurance, multi-currency wallets, FMCG purchases, and micro-withdrawals at provision shops.

The data to do this already exists across remittance, card, wallet, and top-up systems. The problem is that today, finding a cross-sell segment means raising a ticket to the data team, waiting for an extract, and pulling it into a spreadsheet. By the time the campaign is built, the moment has passed.

You are my Streamlit-in-Snowflake engineering partner. Before writing any application code, let's agree on a runtime contract. This app runs inside Snowflake on a managed warehouse runtime that pins a stable, conservative Streamlit baseline, so please adopt the following standards for everything that follows. 1. Session and data access — Obtain the Snowflake session in-process: from snowflake.snowpark.context import get_active_session, then session = get_active_session(). Read data with session.sql(...).to_pandas(). Use this pattern rather than st.connection() for the broadest runtime compatibility. 2. API baseline — Target a conservative Streamlit baseline (assume ~1.22) and use only stable, long-established API surface. Part 3 — Prompt 3.1

Create the semantic view

The business model the agent reasons over
Create a semantic view called MTA_ECOSYSTEM_ANALYST over MERCHANTRADE_LAB.ECOSYSTEM.CUSTOMER_LTV_SEGMENTS for Cortex Analyst. Deploy it with the YAML stored procedure — this is the only reliable path: CALL SYSTEM$CREATE_SEMANTIC_VIEW_FROM_YAML('MERCHANTRADE_LAB.ECOSYSTEM', $$ ...YAML spec... $$); - The first argument is the SCHEMA (database.schema), NOT the view name — the name comes from the YAML's name: field. - Do not use CREATE SEMANTIC VIEW ... AS, YAML =, or SPECIFICATION = — those fail. First run DESCRIBE on the table to get exact column names. In the YAML, write each metric as a table-qualified aggregation, e.g. SUM(CUSTOMER_LTV_SEGMENTS.REMITTANCE_VOLUME). Dimensions: segment, home_state, nationality, acquisition_channel, customer_tier, products_held. Metrics: customer count, total and average remittance volume, total card spend, total wallet top-ups. Add synonyms on segment (e.g. "cross-sell target") and 2-3 verified_queries with real cross-sell questions. Validate first with the procedure's 3rd argument TRUE (dry run), then create it, then confirm with SHOW SEMANTIC VIEWS and one SELECT ... FROM SEMANTIC_VIEW(...) query.
What good looks like
The procedure returns "Semantic view was successfully created", and the verification query returns rows — the governed layer that lets a marketing manager ask "show me cross-sell targets" and get the right answer.
The two rules that prevent the failures
(1) Deploy through SYSTEM$CREATE_SEMANTIC_VIEW_FROM_YAML, not CREATE SEMANTIC VIEW ... AS <yaml> (that DDL doesn't exist). (2) Its first argument is the schema (DB.SCHEMA), not the view name — passing the view name causes "unexpected number of qualifiers". Still no-code as a backup: AI & ML → Cortex Analyst → Create new → Semantic view (Autopilot wizard).
